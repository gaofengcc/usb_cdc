# ESP32 通用 UART 下载工具 — 详细方案

> 聚焦 UART 串口下载场景, 当前优先实现"串口代理学习 + ESP32 回放", 覆盖主流芯片的 Bootloader 烧录。

---

## 1. 覆盖目标芯片（UART Bootloader 类）

| 芯片系列 | 下载协议 | 波特率 | 特殊信号 | 备注 |
|----------|---------|--------|---------|------|
| **ESP32 / ESP8266** | ESP 下载协议 | 115200~921600 | GPIO0 (低电平) + EN (复位脉冲) | 最简单，可自举验证 |
| **STM32 (系统存储器)** | STM32 UART Bootloader | 115200 (默认) | BOOT0=1 + 复位 | AN3155 协议，命令应答式 |
| **NXP LPC** | ISP UART | 9600~115200 | ISP pin + 复位 | 字符握手 |
| **TI CCxx** | Serial Bootloader | 115200 | 专用 IO 触发 | 各系列不同 |
| **GD32 / AT32** | 类 STM32 协议 | 115200 | BOOT0 控制 | 兼容 STM32 命令集 |
| **联发科 / 全志** | UART Download | 115200~921600 | 上电时序特殊 | 私有协议，需学习 |

## 2. 学习引擎核心原理（串口代理优先）

### 为什么当前优先用串口代理而不是 I2S 硬件监听？

当前项目的核心目标不是做通用逻辑分析仪, 而是**尽快稳定拿到 UART 下载协议模板**.  
对于 UART Bootloader, 让下载工具直接连到代理层, 可以直接拿到:

- 上位机下发的 TX 字节流
- 目标芯片返回的 RX 字节流
- 串口参数变更
- RTS/DTR 等控制线变化
- 每帧之间的时间戳

相比 I2S 波形监听, 串口代理的优点是:

- 不需要先做波特率识别和帧同步算法
- 直接拿到字节级协议数据, 模板更稳定
- 更贴近当前仓库已经具备的 `USB CDC ↔ UART` 桥接基础
- 开发和验证闭环更短, 更适合作为 P0 主线

### 串口代理学习链路

```
下载工具 ──虚拟串口/USB CDC──→ 代理层 ──真实 UART──→ 目标芯片
                              │
                              ├── 记录 TX 帧
                              ├── 记录 RX 帧
                              ├── 记录 RTS/DTR
                              ├── 记录 line coding
                              └── 记录时间戳 / 帧间间隔
```

### 代理层需要记录的关键字段

```
session
  ├── uart_config
  │    ├── baudrate
  │    ├── data_bits
  │    ├── parity
  │    └── stop_bits
  ├── control_lines
  │    ├── dtr / rts 变化
  │    └── 变化时间戳
  ├── tx_frames[]
  ├── rx_frames[]
  └── interframe_gap_us
```

### 为什么这已经足够支撑 P0

因为当前覆盖对象主要是标准 UART Bootloader:

- ESP32 / ESP8266 下载协议
- STM32 UART Bootloader
- NXP LPC ISP
- GD32 / AT32 类 STM32 兼容协议

这些协议都以"命令 -> 应答"为主, 使用硬件串口代理即可得到足够准确的模板.

## 3. 回放引擎（下载模式）

### 两种回放策略

#### 策略 A — 按模板字节级回放（当前推荐）

```
代理学习 → 提取命令帧序列 + 应答期望 + 帧间距统计
        → ESP32 按帧用硬件 UART 发送
        → GPIO 回放复位 / Boot 控制
        → 接收目标 ACK, 超时重试
```

- ✅ 数据来源直接是字节流, 精度高
- ✅ 有 ACK 检验
- ✅ 失败时可定位到具体帧
- ✅ 与当前工程的 UART 桥接实现方向一致

**适用于：** 所有带 ACK 的 UART Bootloader, 也是当前 P0 主线

#### 策略 B — I2S 波形学习后回放（后续增强）

```
I2S 学习 → 从波形反推出字节帧 / 控制脉冲
        → 再转成模板化回放
        → 适用于无法接入代理或参数未知的场景
```

- ✅ 支持被动监听
- ✅ 可做疑难时序诊断
- ❌ 需要额外实现采样, 解码, 锁帧算法
- ❌ 当前性价比低于串口代理路线

**适用于：** 参数未知, 不方便修改链路, 或需要波形级诊断的场景

## 4. 信号引脚需求

| ESP32 GPIO | 功能 | 备注 |
|-----------|------|------|
| **USB CDC / UART0** | 上位机入口 | 当前优先作为代理学习入口 |
| **GPIO 10** | UART1 TX → 目标 RX | 下载模式输出 |
| **GPIO 11** | UART1 RX ← 目标 TX | 下载模式接收 ACK |
| **GPIO 12** | 目标 EN/RST | 下载模式控制复位 |
| **GPIO 13** | 目标 BOOT/GPIO0 | 下载模式控制启动模式 |
| **GPIO 14** | LED 指示 | 状态指示灯 |
| **GPIO 15** | 模式按键 | 切换学习/下载模式 |

后续 I2S / IIS 扩展预留:

| 预留 GPIO | 功能 | 备注 |
|----------|------|------|
| **GPIO 4** | I2S 通道 A | 后续被动监听 PC TX |
| **GPIO 5** | I2S 通道 B | 后续被动监听目标 TX |
| **GPIO 6** | RTS 捕获 | 后续监听控制线 |
| **GPIO 7** | DTR 捕获 | 后续监听控制线 |

**电平转换：** 所有 GPIO 通过 TXS0108 双向电平转换芯片引出，支持 3.3V 和 5V 目标芯片。

## 5. MVP 开发计划（2 周）

### 第 1 周 — 串口代理学习链路

| 天 | 任务 |
|---|------|
| D1 | 打通 USB CDC / 虚拟 COM → UART 桥接链路 |
| D2 | 记录 TX/RX 帧, 时间戳和 line coding |
| D3 | 记录 RTS/DTR, 映射为 reset / boot 时序 |
| D4 | 完整学习流程串起来: 导出 JSON 模板 |
| D5 | 用 ESP32 / STM32 实际下载会话验证模板正确性 |

### 第 2 周 — 回放引擎 + 自举验证

| 天 | 任务 |
|---|------|
| D6 | 硬件 UART 按帧回放 + GPIO 复位控制 |
| D7 | ACK 接收 + 超时重试逻辑 |
| D8 | 模式切换（学习/下载）+ LED + 按键交互 |
| D9 | **自举测试：用样板 ESP32 去烧另一个 ESP32** |
| D10 | 优化 + 记录 BUG 和坑 |

### 后续阶段 — I2S / IIS 被动监听增强

当 P0 主链路稳定后, 再评估以下工作:

| 阶段 | 任务 |
|---|------|
| P2-1 | I2S 并行采样初始化 + 波形捕获验证 |
| P2-2 | 波特率自动识别算法 + 数据帧提取 |
| P2-3 | 帧同步验证 + RTS/DTR 脉冲识别 |
| P2-4 | 旁路学习模式与模板导出整合 |

## 6. 第一代产品规格

```
ESP32 Universal UART Flasher v1

硬件：ESP32-S3 + TXS0108 + SD卡 + USB-C + 按键x1 + LEDx1
尺寸：约 60x30mm（比火柴盒略大）
接口：6pin 排针（VCC/TX/RX/RTS/DTR/GND）

支持协议：
  - 标准 UART（7/8 bit, 偶/奇/无校验, 1/2 stop bit）
  - 代理学习模式下直接记录 UART 参数
  - RTS/DTR 自动复位信号学习+回放

工作模式：
  ① LEARN — 串口代理学习, 记录下载交互
  ② FLASH — 接入目标芯片 TX/RX，直接下载
  ③ PASS — 透传模式, 当 USB↔UART 转接板用
  ④ SNIFF — 后续 I2S 被动监听模式

存储：SD卡存储多个协议模板，支持命名和选择

供电：USB-C 5V 或目标板取电（3.3V/5V）
```
